# doks-flux-demo

A minimal, end-to-end DOKS + Flux + Tailscale demo for intermediate IT folks who want to see what a private GitOps cluster looks like without signing up for half a dozen services.

You fork this repo, fill in three environment variables, run `mise run create`, and ~10 minutes later you have a 2-node DigitalOcean Kubernetes cluster with Flux pulling from your fork and a sample app (uptime-kuma) reachable privately on your tailnet.

## What you'll build

```
                                +------------------+
              GitHub fork  <----+   Flux           |
              (this repo)       |   (in-cluster)   |
                                +--------+---------+
                                         |
                                         v
+----------------+   +---------------------+   +-------------------+
| Your tailnet   |<--| Tailscale operator  |<--| DOKS cluster       |
| (private)      |   | (Helm via Flux)     |   | 2 x s-2vcpu-2gb    |
+--------+-------+   +---------------------+   +---------+---------+
         |                                               |
         v                                               v
   uptime-kuma-cloud.<your-tailnet>.ts.net         StatefulSet + PVC
```

- **DigitalOcean** runs the cluster control plane and worker nodes.
- **Flux** runs in-cluster and reconciles manifests from your fork.
- **Tailscale operator** publishes Ingress objects as private hosts on your tailnet -- no public LB, no DNS, no certs to manage.
- **uptime-kuma** is the demo workload: a SQLite-backed status page on a 5Gi PVC.

## Cost

About $36/month while running:

- Control plane: free
- 2 x `s-2vcpu-2gb`: ~$36/mo
- Block storage (5Gi): ~$0.50/mo

`mise run destroy` removes everything.

## Prereqs

Accounts:

- DigitalOcean (with billing enabled)
- Tailscale (free personal tier is fine)
- GitHub (this repo lives in your fork)

Tailscale ACL tags (one-time setup):

The Tailscale operator needs two tags to exist in your tailnet's Access Controls before you create the OAuth client. Edit your ACL at <https://login.tailscale.com/admin/acls/file> and ensure the `tagOwners` block contains:

```jsonc
"tagOwners": {
  "tag:k8s-operator": ["autogroup:admin"],
  "tag:k8s":          ["tag:k8s-operator"],
}
```

`tag:k8s-operator` is the operator's own identity. `tag:k8s` is what Tailscale-managed proxy devices wear when the operator publishes Ingress objects.

CLI tools:

Install [mise](https://mise.jdx.dev), then run `mise run bootstrap` from the repo root. This installs every CLI this repo uses (`tofu`, `terragrunt`, `doctl`, `kubectl`, `flux`, `gh`, `uv`) at the versions recorded in `mise.lock` and syncs the Python deps into `.venv`. Run `mise run versions` afterwards to confirm.

You'll still need `git` (to clone) and an authenticated `gh` (run `gh auth login` once after bootstrap — `flux bootstrap` uses it).

## Setup

```bash
# 1. Fork this repo to your GitHub account, then clone the fork.
git clone https://github.com/<you>/doks-flux-demo.git
cd doks-flux-demo

# 2. Configure secrets.
cp .env.example .env
# Edit .env and fill in:
#   DIGITALOCEAN_TOKEN          (DO control panel -> API)
#   TS_OPERATOR_CLIENT_ID       (Tailscale -> Settings -> OAuth clients)
#   TS_OPERATOR_CLIENT_SECRET

# 3. Install tools and the CLI.
mise run bootstrap

# 4. Authenticate the GitHub CLI (used by `flux bootstrap`).
gh auth login

# 5. Provision.
mise run create
```

`mise run create` will:

1. Run `terragrunt plan`, show you a summary, and ask before applying.
2. Apply: create a DOKS cluster, node pool, and DO project.
3. Save kubeconfig via `doctl`.
4. Run `flux bootstrap github` against your fork (creates a deploy key, commits the Flux components).
5. Apply the Tailscale operator OAuth secret directly from `.env`.
6. Reconcile `infra-edge` (Tailscale operator) and `apps-uptime-kuma`.
7. Print the tailnet URL.

When it's done, open `https://uptime-kuma-cloud.<your-tailnet>.ts.net` in a browser (you must be on your tailnet).

## Optional: enable backups

The `uptime-kuma-cloud` app ships with a backup CronJob that takes daily DigitalOcean VolumeSnapshots of the PVC (no external storage, no extra credentials). Backups are off by default.

To enable:

1. Edit `kubernetes/clusters/doks/kustomization.yaml` and uncomment the `apps-uptime-kuma-backups.yaml` line.
2. Commit and push to your fork.
3. Wait for Flux to reconcile (or run `flux reconcile source git flux-system`).

That edit-commit-push-reconcile loop *is* the GitOps demo.

Verify:

```bash
kubectl -n uptime-kuma-cloud get cronjob
kubectl -n uptime-kuma-cloud create job --from=cronjob/uptime-kuma-cloud-backup manual-1
kubectl -n uptime-kuma-cloud get volumesnapshot
```

## Operate

```bash
# Health check
mise run status

# Reconcile from main
flux reconcile source git flux-system
flux reconcile kustomization apps-uptime-kuma --with-source

# Direct cluster access
kubectl get pods -A
flux get kustomizations -A
```

## Teardown

```bash
mise run destroy
```

This will:

1. Suspend Flux and delete the app + infrastructure Flux Kustomizations (so finalizers can clean up the Tailscale machine and the DO Load Balancer if any).
2. Run `terragrunt destroy`.
3. Audit for orphaned DO resources tagged `k8s:<cluster-uuid>` (load balancers, volumes, firewalls) and delete them.
4. Delete the empty `doks-flux-demo-project` DigitalOcean project.
5. Delete offline tailnet devices created by this demo (the `doks-flux-demo-operator` device and `uptime-kuma-cloud*` proxies). Online devices with similar names are skipped, so a co-tenant cluster on the same tailnet is safe.

Flux deploy keys on your GitHub fork are not auto-removed -- clean them up at `https://github.com/<you>/doks-flux-demo/settings/keys`.

## What's intentionally not here

This repo is a teaching demo, not a platform template. It deliberately omits:

- **External Secrets Operator** -- secrets come from `.env`, not a vault.
- **cert-manager / public ingress / DNS** -- everything is private via Tailscale.
- **Multi-app structure** -- one app keeps the Flux Kustomization graph readable.
- **Restic / B2 / off-cluster backups** -- VolumeSnapshots stay inside DigitalOcean.
- **HA control plane, autoscaling, multiple node pools** -- one pool, two nodes.
- **CI** -- bootstrap is a developer-laptop workflow.

The companion homelab repo this was extracted from has all of those, if you want to see the production shape.

## Customize

The two knobs you'll likely want to change live in `tofu/deployments/doks/terragrunt.hcl`:

- `cluster_name` -- DigitalOcean cluster name (default: `doks-flux-demo`). Change this if the default would collide with another cluster on your account.
- `local.region` -- DigitalOcean region slug (default: `sfo3`). `vpc_name` derives from this automatically (`default-${local.region}`).

The CLI reads `cluster_name` and `project_name` from this file at runtime, so editing terragrunt.hcl is the single point of change. Other values (`node_size`, `node_count`, `project_name`, tags) are intentionally opinionated to keep the demo small; edit them in the same file if you must.

`.env` is for **secrets only** -- never put cluster identity or region there.

## Troubleshooting

```bash
# Cluster check
doctl kubernetes cluster list
kubectl get nodes -o wide

# Flux state
flux get sources git -A
flux get kustomizations -A
flux get helmreleases -A
flux logs --all-namespaces --follow=false

# Tailscale operator
kubectl -n tailscale get pods
kubectl -n tailscale get secret operator-oauth
kubectl -n tailscale logs deploy/operator

# uptime-kuma
kubectl -n uptime-kuma-cloud get pods,svc,ingress
```

## Layout decisions worth knowing about

- **Local Tofu state.** State lives in `tofu/deployments/doks/.terragrunt-cache/`. Run `mise run destroy` before wiping your laptop, or back up the state files. For real clusters, configure a remote backend.
- **One Flux path.** `flux bootstrap` points at `kubernetes/clusters/doks/`; that path's `kustomization.yaml` is the entry point.
- **Tailscale ingress is private-only.** There is no DNS to configure -- Tailscale's MagicDNS handles it. You must be on your tailnet to reach the app.
- **Secrets in `.env`** are read at every CLI invocation and passed to subprocesses as environment variables. Nothing is written to disk by the CLI.
